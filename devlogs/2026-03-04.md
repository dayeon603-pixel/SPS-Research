Built `StructuredSensitivityEstimator`:

```python
class StructuredSensitivityEstimator:
    def estimate(
        self,
        embeddings: torch.Tensor,       # (B, seq_len, hidden_dim)
        input_ids: torch.Tensor,        # (B, seq_len)
        attention_mask: torch.Tensor,   # (B, seq_len)
        m: int = 32,
    ) -> torch.Tensor:                  # (B,) — per-sample sensitivity
```

Core loop: for each of $m$ transformations, sample $(T_x, c_x)$ from the transform family, forward-pass both original and perturbed through the model, compute divergence, divide by $c_x$, accumulate max over $m$.

**`src/sps/transformations.py` — three families:**

`EmbeddingPerturbationFamily` ($\mathcal{T}_{\text{emb}}$): continuous, differentiable. $T_\alpha^v(E) = E + \alpha v$ where $v \in A_x$ is a synonym-derived unit direction.

```python
class EmbeddingPerturbationFamily(TransformationFamily):
    def sample(
        self,
        embeddings: torch.Tensor,
        input_ids: torch.Tensor,
        epsilon: float,
    ) -> tuple[torch.Tensor, torch.Tensor]:  # (perturbed, magnitudes)
```

`SynonymSubstitutionFamily` ($\mathcal{T}_{\text{syn}}$): discrete token replacement via precomputed synonym map. Not differentiable — cannot be used for JVP computation.

`AdversarialEmbeddingFamily` ($\mathcal{T}_{\text{adv}}$): worst-case direction within $A_x$, found by maximizing $\|J_f(x)\, v\|$ over all synonym directions.

Abstract base `TransformationFamily` defines the interface:
- `.sample(embeddings, input_ids, epsilon) -> (perturbed, magnitudes)`
- `.semantic_directions(embeddings, input_ids) -> (B, K, seq, hidden)`

**Memory: EmbeddingPerturbationFamily with `use_synonym_directions=True`:**

Building synonym directions for the full BERT vocabulary (30,522 tokens) takes ~3.7 GB RAM at import time. This will be a problem. Filed as something to fix — will add a vocab cap. (See devlog 2026-03-24 for the eventual fix.)

**JVP on BERT-base (12 layers, 768 hidden, seq_len=128):**

One JVP call takes ~18ms on CPU. For n=512 samples, m=32 transforms: $512 \times 32 \times 18\text{ms} \approx 295\text{s}$ for the full JVP-based computation. The sampling-based estimator (using actual model forward passes) is slightly slower at ~22ms per forward pass pair, giving ~361s. So JVP is faster but not dramatically so on CPU. On GPU the gap would be larger.

### Lab Notes

The design tension in `StructuredSensitivityEstimator._forward()` is real. The model needs to accept `inputs_embeds` instead of `input_ids` so we can pass perturbed embeddings directly. HuggingFace models support this but the interface is slightly different across architectures. Built a fallback chain that tries multiple attribute paths — inelegant but necessary.

The `AdversarialEmbeddingFamily` requires a `forward_fn` parameter at construction time, which creates a circular dependency risk if the same model is used as both the evaluation target and the adversarial direction finder. That's technically fine (the JVP of the model used to find the worst direction is the same as the model being evaluated) but it feels worth flagging. The adversarial family is measuring how hard you can hit the model's own weaknesses — which is what you want.

Gram-Schmidt in `_random_orthogonal_directions` runs on CPU even when the rest of the computation is on GPU — it's small (K=8, hidden=768) and not a bottleneck, but should be pushed to device later.

### Hypotheses

1. **The WordNet synonym map memory issue will need a vocab cap of ~10k tokens.** Full vocab at 50k is too expensive. Most tokens outside the top 10k are rare enough that they won't appear in typical evaluation data anyway. Cap at 10k should cover >95% of typical NLP evaluation examples.

2. **JVP-based SPS computation will be 3–5x faster than sampling-based on GPU.** The JVP computes sensitivity in one forward pass per direction. The sampling estimator requires two forward passes (original + perturbed). On GPU with large batch sizes, the JVP advantage should grow.

3. **SPS estimates from m=32 transforms are within 5% of m=128 estimates.** Beyond a certain m, adding more transformations gives diminishing returns. Will check the convergence of the estimator as m increases.
