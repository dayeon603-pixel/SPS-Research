**Read:** GitHub documentation on math expressions in Markdown. Read several GitHub issues on MathJax/KaTeX rendering failures. Read CommonMark specification section on emphasis and strong emphasis.

Key finding from GitHub docs: GitHub uses a modified CommonMark pipeline. Math expressions (`$...$` inline, `$$...$$` display) are handled by KaTeX. The pipeline is sequential: CommonMark parsing first, then KaTeX rendering on the resulting AST nodes.

Critical vulnerability: inside blockquote syntax (`> ...`), CommonMark's emphasis detection is active. The parser sees `_` as a potential italic marker before KaTeX sees it as a subscript.

Concrete failure modes:
1. `> $\mathrm{Sens}_{...}$` — the `_` in `_{...}` inside a blockquote can trigger italic mode
2. `**$f_\theta$**` — `_` inside bold context can also cause issues
3. Any LaTeX command with `_` inside a Markdown element that processes emphasis

**Prediction:** My original README draft used blockquotes for definition callouts. The formulas with `_{\mathcal{T},\varepsilon}` subscripts will break inside blockquotes.

## [17:03 UTC] Implementation Session — README Rendering Bug Found and Fixed

### State at Session Start

README written with blockquote-based definition callouts. Pushed to GitHub. Opened the page to check rendering.

### Work Completed

**The bug I found on first push:**

Definition 1 with subscript `_{\mathcal{T},\varepsilon}` inside a blockquote:

```markdown
> **Definition 1 (Structured Local Sensitivity).**
> For $\varepsilon > 0$:
>
> $$\mathrm{Sens}_{\mathcal{T},\varepsilon}(f_\theta;\, x) \;:=\; \sup_{\substack{T \in \mathcal{T} \\ 0 < c(T,x) \leq \varepsilon}} \frac{d_\mathcal{Y}\!\bigl(f_\theta(x),\, f_\theta(Tx)\bigr)}{c(T,x)}$$
```

Rendered as:

```
Sens_{T ,*ε*}(fθ; x) := sup ... broken mixed output
```

The `_{\mathcal{T},\varepsilon}` subscript was partially parsed as italic. CommonMark sees `_{\mathcal{T},` as starting italic mode, closes it somewhere inside the LaTeX, and the result is a mix of italicized fragments and broken math symbols.

**Root cause analysis:**

GitHub's Markdown rendering pipeline is sequential, not parallel:
1. CommonMark parsing: processes blockquotes, emphasis, links, etc.
2. KaTeX rendering: processes `$...$` nodes left by the first pass.

Inside a blockquote, the parser is more aggressive about italic detection. The `_` in `_{\mathcal{T},\varepsilon}` is ambiguous — it could start italic mode. CommonMark's emphasis algorithm resolves the ambiguity by treating it as italic. The math renderer then receives malformed LaTeX.

**The fix:**

Removed all blockquotes from math-containing sections. Moved to standalone display math with blank lines:

```markdown
**Definition 1 (Structured Local Sensitivity).**
For $\varepsilon > 0$:

$$\mathrm{Sens}_{\mathcal{T},\varepsilon}(f_\theta;\, x) \;:=\; \sup_{\substack{T \in \mathcal{T} \\ 0 < c(T,x) \leq \varepsilon}} \frac{d_\mathcal{Y}\!\bigl(f_\theta(x),\, f_\theta(Tx)\bigr)}{c(T,x)}$$
```

Pushed. GitHub renders correctly.

**Systematic review of all render-breaking patterns:**

| Pattern | Problem | Fix |
|---|---|---|
| `> $$...$$` | Blockquote + display math | Remove blockquote |
| `**$f_\theta$**` | Bold wrapping inline math with `_` | Don't wrap inline math with bold |
| `> $\mathrm{Sens}_{\mathcal{T},\varepsilon}$` | Blockquote + inline math with multiple `_` | Remove blockquote |
| LaTeX in table cell with `_` | Usually fine if no italic context | Keep, verify per cell |

Applied fix systematically: the assumptions A3–A5 had also been inside blockquotes. Moved to flat text with bolded labels.

Added `CONTRIBUTING.md` note:

```markdown
## Math rendering
Always test LaTeX rendering on GitHub's web preview before committing.
Never put display math inside blockquote (>) syntax — CommonMark processes
italic markers before handing off to the math renderer. Use standalone
$$...$$ blocks with blank lines on both sides.
```

Committed as `fix(readme): remove blockquotes from math definitions, standalone display math`.

**Reflection on the failure mode:**

This is the kind of error that's invisible in local editors (VS Code renders LaTeX correctly inside blockquotes) and only visible on the target platform. The GitHub pipeline is not what any local editor uses. Always render-test on the actual deployment environment.

The fix cost ~20 minutes. It would have cost 0 minutes if I had tested on GitHub before pushing. "Test on the actual platform" is the same lesson as "test in production conditions" in software.
