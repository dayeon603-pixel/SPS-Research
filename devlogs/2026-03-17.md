Pushed. Opened GitHub to check rendering. The formula for $\mathrm{Sens}_{\mathcal{T},\varepsilon}$ looked like:

```
Sens_{T,ε}(fθ; x) := supSens_{T ,*ε*}
```

Broken. The subscript `_{\mathcal{T},\varepsilon}` was being partially parsed as italic markup because of the underscores. Specifically `_{\mathcal{T},\varepsilon}` → the opening `_` started italic mode, and the parser closed it somewhere inside the LaTeX, producing a mix of italicized text and broken math.

The problem is that GitHub's CommonMark renderer processes the Markdown italic/bold markers before passing to the math renderer. Inside a blockquote (which is also a Markdown construct), the parsing is even more aggressive. The `_` in `_{\mathcal{T},\varepsilon}` is ambiguous — Markdown sees it as potential italic syntax before the math renderer sees it as a subscript.

**The fix: remove blockquotes, put math on standalone lines.**

Changed to:

```markdown
**Definition 1 (Structured Local Sensitivity).**
For $\varepsilon > 0$:

$$\mathrm{Sens}_{\mathcal{T},\varepsilon}(f_\theta;\, x) \;:=\; \sup_{\substack{T \in \mathcal{T} \\ 0 < c(T,x) \leq \varepsilon}} \frac{d_\mathcal{Y}\!\bigl(f_\theta(x),\, f_\theta(Tx)\bigr)}{c(T,x)}$$
```

Standalone display math (`$$...$$`) on its own line, no blockquote wrapping. The Markdown parser treats the entire `$$...$$` block as a math node and passes it to the math renderer without trying to interpret the contents as Markdown.

Pushed. GitHub rendered correctly.

### Lab Notes

The blockquote + math failure is not documented anywhere I could find. The GitHub math rendering documentation just says to use `$...$` for inline and `$$...$$` for display. It doesn't mention the blockquote interaction.

The root cause: GitHub's Markdown pipeline is sequential, not parallel. CommonMark parsing (including italic/bold) happens in one pass, then MathJax/KaTeX rendering happens in a second pass on the `$...$` nodes. Blockquotes are processed in the CommonMark pass, and the parser sees `_` inside a blockquote as potential italic syntax, which it resolves before the math pass gets to see the LaTeX.

The fix is to structure the Markdown so that math-heavy content is never inside any Markdown construct that uses `_` as a special character. Blockquotes are the main danger zone. Emphasis markers (`*...*` or `_..._`) are another.

Counterintuitively, LaTeX in raw HTML `<div>...</div>` or `<span>...</span>` blocks also sometimes breaks depending on the renderer. The safest approach is standalone display math with blank lines on both sides.

This is the kind of thing that costs 20 minutes and feels embarrassing in retrospect. The fix is 10 lines. The lesson is to always render-test math-heavy Markdown on the actual target platform before finalizing.

### Hypotheses

1. **All remaining `$$...$$` blocks in the README are render-correct after the fix.** The systematic review covered all display math. Inline math without ambiguous underscores should also be fine.

2. **The blockquote rendering issue also affects GitLab, Bitbucket, and other CommonMark renderers.** The pipeline (CommonMark first, then math) is common. Standalone display math is the safe universal approach.
