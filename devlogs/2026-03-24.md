### Work Completed

**Root cause analysis.**

`build_wordnet_synonym_map()` in `src/sps/transformations.py` was called at `EmbeddingPerturbationFamily` construction time with `vocab_size=None`, defaulting to `tokenizer.vocab_size` (30,522 for BERT, 50,265 for RoBERTa). For each token in the vocabulary:
1. Decode to string
2. Look up WordNet synsets
3. For each lemma, encode back to token

30,522 WordNet lookups at construction time, building a dict of size ~8,400 entries (the tokens that actually have synonyms). Total memory: dict overhead + embedding lookups = ~3.9 GB peak.

The silent fallback was in `EmbeddingPerturbationFamily.semantic_directions()`:

```python
if self.config.use_synonym_directions and self.synonym_map:
    return self._synonym_directions(embeddings, input_ids, K)
else:
    return self._random_orthogonal_directions(B, seq_len, hidden, K, device)
```

The `self.synonym_map` check was a truthiness check. For tokens not in the map (beyond the RAM limit, where the map construction was killed early by OOM), the fallback silently triggered. The logger call inside `_synonym_directions` was at DEBUG — invisible in default logging config.

**Fix 1: Vocabulary cap.**

Added `vocab_size` parameter to `build_wordnet_synonym_map()` with default 10,000:

```python
def build_wordnet_synonym_map(
    tokenizer,
    vocab_size: Optional[int] = None,
    max_synonyms_per_token: int = 5,
) -> dict[int, list[int]]:
    vocab_size = vocab_size or tokenizer.vocab_size
    # ... (unchanged logic, but now respects the cap)
    logger.info(
        "Built WordNet synonym map: %d / %d tokens have synonyms.",
        len(synonym_map),
        vocab_size,
    )
    return synonym_map
```

At `vocab_size=10000`: ~1.1 GB peak RAM, ~8.3 seconds construction time.
At `vocab_size=50000`: ~4.2 GB peak RAM, ~41 seconds construction time.

The top 10,000 most frequent tokens in BERT's vocabulary cover >95% of typical English text. Tokens beyond rank 10,000 are rare — they wouldn't appear in SST-2 evaluation data with meaningful frequency.

**Fix 2: Explicit WARNING on fallback.**

In `EmbeddingPerturbationFamily._synonym_directions()`:

```python
for b in range(B):
    for s in range(seq_len):
        tid = input_ids[b, s].item()
        candidates = self.synonym_map.get(int(tid), [])
        if not candidates:
            fallback_count += 1
            # use original token (direction will be near-zero + noise)
```

Added a per-batch warning when fallback rate exceeds 20%:

```python
if fallback_count / (B * seq_len) > 0.2:
    logger.warning(
        "Synonym fallback rate %.1f%% (>20%%) — synonym_map may be incomplete. "
        "Directions for unmapped tokens use noise only.",
        100.0 * fallback_count / (B * seq_len),
    )
```

**Fix 3: Lazy loading.**

Moved synonym map construction out of `__init__` — now done lazily on first call to `semantic_directions()`. Prevents import-time RAM spike:

```python
class EmbeddingPerturbationFamily(TransformationFamily):
    def __init__(self, ...) -> None:
        self.synonym_map = synonym_map or {}
        self._synonym_map_loaded: bool = bool(synonym_map)

    def _ensure_synonym_map(self, tokenizer, vocab_size: int = 10_000) -> None:
        if not self._synonym_map_loaded:
            self.synonym_map = build_wordnet_synonym_map(tokenizer, vocab_size=vocab_size)
            self._synonym_map_loaded = True
```

Note: the lazy loading only applies when `synonym_map=None` is passed at construction. If a pre-built map is passed, it's used immediately (existing behavior preserved).

### Lab Notes

This was a silent correctness bug. The preliminary experiments looked plausible because the random orthogonal directions produced plausible sensitivity values — they're not zero, they're not infinity, they're just wrong. The fallback gave numbers that fell in the expected range, which is exactly what makes silent bugs hard to catch.

The fix that mattered most was the WARNING log. The vocab cap and lazy loading are quality-of-life improvements, but they don't actually prevent the fallback from happening on rare vocabulary. The WARNING at >20% fallback rate is what makes the fallback visible.

Lesson: silent fallbacks with plausible-looking outputs are the worst kind of bug. Any time code silently substitutes a default or fallback behavior, it should log at WARNING or ERROR, not DEBUG or INFO.

Also: any experiments run before this fix (specifically the Mar 20 comparison) need to be flagged as preliminary in the paper. The corrected values are within 0.02 of the preliminary values, so the qualitative conclusions are the same — but the exact numbers in the paper should be from the corrected run.

The memory profile is interesting: construction time is roughly linear in vocab_size (8.3 s for 10k, 41 s for 50k). This is consistent with $O(V)$ WordNet lookups. The RAM usage is also roughly linear. There's no clever way to make this sublinear — each token needs a separate lookup.

### Hypotheses

1. **`vocab_size=10000` covers >95% of tokens in standard English NLP benchmarks.** Zipf's law: top 10k types account for most tokens in any large natural language corpus. For SST-2, MNLI, SQuAD — datasets drawn from published English text — this holds.

2. **The 0.02 SPS decrease from corrected synonym directions vs random fallback is consistent across models and datasets.** The preliminary minus corrected gap is ~0.02 for both BERT and RoBERTa. This suggests that random orthogonal directions give slightly higher SPS than synonym directions (synonym directions are semantically loaded and therefore harder). The gap should be model-independent to first order.

3. **Re-running any experiments from before Mar 24 with the fix will shift SPS values down by approximately 0.01–0.03.** This is small enough that qualitative conclusions won't change, but the exact numbers in tables need to use corrected runs.
